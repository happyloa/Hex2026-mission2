金融產品的設計難度，往往不在於畫面是否漂亮，而在於如何把大量、敏感且即時變動的資料轉換成使用者能理解並信任的介面。當 Open Bank API 讓更多金融資料得以整合，設計師與工程師也必須一起思考：資料如何呈現、錯誤如何回饋、安全性如何被感知。

## 複雜資料需要先被整理成決策線索

帳戶餘額、交易紀錄、信用狀態、投資比例，這些資料對系統來說只是欄位，對使用者來說卻是生活決策的依據。金融 App 的介面不應該把所有資訊一次丟出來，而是透過分層、摘要與視覺化，讓使用者快速掌握目前狀態，再進一步查看細節。

## API 串接要設計異常狀態

Open Bank API 的資料來源可能來自不同金融機構，因此延遲、授權過期、欄位不一致都可能發生。如果介面只設計成功狀態，產品上線後就會充滿破碎體驗。設計階段就應該納入 loading、partial data、retry、授權重新連結等流程，讓使用者知道問題發生在哪裡，以及下一步可以做什麼。

## 安全性需要被使用者看見

後端安全機制再完整，如果使用者無法理解，也很難建立信任。例如連結銀行帳戶前的權限說明、資料更新時間、登出與撤銷授權的位置，都會影響使用者是否敢繼續操作。這些不是附屬文案，而是金融體驗的重要部分。

## AI 可以協助辨識異常與摘要資訊

在金融資料越來越龐大的情境中，AI 可以協助偵測異常交易、整理消費趨勢，甚至提供財務建議。但這類功能必須非常謹慎地設計解釋方式，避免讓使用者誤以為系統擁有絕對判斷。AI 的角色應該是輔助理解，而不是替使用者做不可逆決策。

| 設計挑戰     | 技術對應              | 體驗目標         |
| ------------ | --------------------- | ---------------- |
| 資料量龐大   | 摘要 API 與快取策略   | 快速掌握重點     |
| 授權流程複雜 | OAuth 與權限分層      | 清楚理解資料用途 |
| 異常狀態頻繁 | 錯誤分類與 retry 機制 | 降低焦慮         |
| 資安信任不足 | 操作紀錄與撤銷授權    | 建立控制感       |

金融 App 的設計是一場信任工程。當資料呈現、後端架構與介面回饋能彼此配合，使用者才會願意把重要的財務決策交給產品協助。
