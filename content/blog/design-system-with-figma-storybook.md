設計系統真正困難的地方，不是做出一套漂亮的元件庫，而是讓它在團隊日常中持續被使用。當產品規模變大，設計稿、程式碼與文件只要有一方開始脫節，介面就會慢慢長出不一致的細節。Figma 與 Storybook 的搭配，正是我在多年專案中覺得最能兼顧美感與工程邏輯的工作方式。

## 先定義 token，再討論元件

很多團隊一開始就急著整理 Button、Card、Modal，卻忽略顏色、字級、間距與狀態這些基礎 token。結果元件看似完整，實際開發時仍然充滿例外。我的做法是先把品牌色、文字階層、圓角、陰影與互動狀態整理成可命名的 token，讓設計與工程對同一個語言有共識。

## Figma 負責視覺語言，Storybook 負責真實狀態

Figma 適合定義元件的視覺規格與使用情境，但真實產品裡的元件會遇到更多狀態：長文字、空資料、載入中、錯誤、disabled、hover、focus。Storybook 的價值就在於把這些狀態用可互動的方式記錄下來，讓設計師、工程師與 PM 都能在同一個頁面討論。

## 命名規則決定維護成本

一套設計系統如果命名混亂，後期會變成團隊負擔。顏色不應只叫 orange 或 gray，而應該對應到語意，例如 primary、neutral、danger。文字階層也應該用 display、heading、paragraph、label 這類能描述用途的命名，讓 utility class 與設計稿能自然對齊。

## 文件化不是額外工作，而是交付的一部分

企業級設計系統需要的不只是元件本身，還要清楚說明何時使用、何時不要使用，以及有哪些可調整的 props。這些文件可以讓新成員快速進入狀況，也能降低跨團隊溝通成本。

| 系統層級      | 工具角色                          | 交付成果             |
| ------------- | --------------------------------- | -------------------- |
| Design Tokens | Figma Variables / Tailwind Config | 色彩、字級、間距命名 |
| Components    | Figma Components / Vue Components | 可重用元件與狀態     |
| Documentation | Storybook                         | 使用情境與互動測試   |
| Governance    | Review 流程                       | 新增與修改規則       |

設計系統不會在第一次建立時就完美。它需要被專案反覆驗證，再從真實使用情境中修正。當設計與工程願意共同維護它，它就會成為產品長期穩定成長的基礎。
